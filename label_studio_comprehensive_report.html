مقدمة المشروع وتحليل الهيكلة

مشروع Label Studio هو منصة مفتوحة المصدر لتصنيف البيانات. يعتمد المشروع على تقنيات حديثة ولكنه يواجه تحديات عند استخدامه في بيئات إنتاجية ضخمة أو مع بيانات حساسة.

المكونات الرئيسية:

المكون التقنية المستخدمة الوظيفة
الواجهة الخلفية (Backend) Django / Python إدارة الـ API، قاعدة البيانات، والمستخدمين.
الواجهة الأمامية (Frontend) React / TypeScript واجهة التصنيف التفاعلية وإدارة المهام.
قاعدة البيانات PostgreSQL / SQLite تخزين المشاريع، المهام، والتحديدات.
ملاحظة: المشروع مصمم بنظام "Mixins" مما يجعله مرناً جداً للتعديل، ولكن هذا يتطلب فهماً عميقاً لكيفية وراثة الكلاسات داخل Django.

التحليل الأمني العميق (Security Deep Dive)

تم فحص ملف label_studio/core/settings/base.py والملفات المرتبطة به، وتم استخراج الثغرات التالية:

1. ثغرة ALLOWED_HOSTS (حرجة)

الكود الحالي يسمح بجميع الـ Hosts افتراضياً:

settings/base.py
ALLOWED_HOSTS = get_env_list('ALLOWED_HOSTS', default=['*'])

الخطر: يسمح بهجمات Host Header Injection. المهاجم يمكنه خداع السيرفر لإنشاء روابط خبيثة (مثل روابط استعادة كلمة المرور) ترسل لبريد المستخدم ولكنها تشير لسيرفر المهاجم.

2. تعطيل حماية CSRF (خطير)

يوجد Middleware مخصص لتعطيل الحماية:

core/middleware.py
class DisableCSRF(MiddlewareMixin):
  def process_view(self, request, callback, *args, **kwargs):
    setattr(request, '_dont_enforce_csrf_checks', True)

الخطر: هذا يفتح الباب لهجمات Cross-Site Request Forgery. إذا قام المستخدم بزيارة موقع خبيث وهو مسجل دخول في Label Studio، يمكن للموقع الخبيث إرسال طلبات (مثل حذف بيانات) دون علم المستخدم.

3. ثغرة CORS (Cross-Origin Resource Sharing)

الإعداد الافتراضي يسمح للجميع:

settings/base.py
CORS_ALLOW_ALL_ORIGINS = get_bool_env('CORS_ALLOW_ALL_ORIGINS', True)

الخطر: يسمح لأي موقع على الإنترنت بقراءة بيانات الـ API الخاصة بك إذا كان المستخدم مسجلاً للدخول.

4. حماية SSRF المعطلة

الحماية من طلبات السيرفر المزيفة معطلة افتراضياً:

settings/base.py
SSRF_PROTECTION_ENABLED = get_bool_env('SSRF_PROTECTION_ENABLED', False)

الخطر: يمكن للمهاجم استخدام خاصية "Import from URL" لجعل السيرفر يهاجم نفسه أو يكتشف خدمات داخلية في شبكتك الخاصة.

تحليل الأداء والضغط (Performance Analysis)

لماذا يثقل النظام عند زيادة النقوش الفرعونية؟

أ. مشكلة الـ JSON Serialization

يتم تخزين التحديدات في حقل JSONField. عند طلب صورة تحتوي على 500 نقش، يقوم Django بتحويل هذا الـ JSON الضخم إلى Object في كل مرة. هذا يستهلك الـ CPU بشكل هائل.

ب. مشكلة قاعدة البيانات (SQLite vs PostgreSQL)

إذا كان العميل يستخدم SQLite، فإن أي عملية كتابة (Save Label) تقوم بقفل قاعدة البيانات بالكامل (Database Lock)، مما يمنع المستخدمين الآخرين من القراءة أو الكتابة حتى تنتهي العملية.

ج. رندر الواجهة الأمامية (Frontend Rendering)

واجهة React تستخدم Canvas أو SVG لرسم التحديدات. مع زيادة عدد النقوش، تزداد العمليات الحسابية في المتصفح، مما يسبب "تهنيج" في جهاز المستخدم.

نظام الأدوار والصلاحيات (RBAC)

النسخة المجتمعية تفتقر لنظام أدوار حقيقي. إليك التحليل الفني لكيفية إضافته:

الموديلات المتأثرة:

  • User Model: يحتاج لحقل role.
  • ProjectMember Model: يحتاج لتحديد صلاحيات العضو داخل المشروع.

الأدوار المقترحة للعميل:

الدور الصلاحيات
مدير (Admin) كامل الصلاحيات (إنشاء مشاريع، حذف، إدارة مستخدمين).
مُراجع (Reviewer) رؤية جميع الصور وتعديل التحديدات، لكن لا يمكنه حذف المشروع.
مُصنِّف (Annotator) رؤية الصور المخصصة له فقط، ولا يمكنه رؤية عمل الآخرين.
